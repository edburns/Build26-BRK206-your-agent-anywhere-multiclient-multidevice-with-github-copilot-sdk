Root cause: The `copilot --yolo` session has an idle timeout. Once the polling command exceeded `initial_wait: 120s` and was backgrounded, the agent said "I'll check back when the polling completes" and went idle. The `assistant.idle` event fired at 00:25:45, and ~66 seconds later the session was killed — even though the background command was still running and actually found the review.

This is an externally imposed idle timeout, not a skill bug. The `--yolo` mode appears to terminate the session shortly after the agent goes idle (no pending tool calls from the agent's perspective). The background shell command was still running, but the runtime treats the agent as "done" once it's idle.

Contrast with BRK206-01: In that run, sessions lasted 22 min to 10+ hours. Either the idle timeout didn't exist in that CLI version, or the agent structured its polling differently (not backgrounding the command).

Fix options:

1. Have the agent use a shorter `initial_wait` that doesn't exceed the idle timeout — but then it needs to actively loop itself rather than relying on background commands
2. The skill should instruct the agent to never go idle while waiting — instead of backgrounding the poll, it should block on it with a long `initial_wait` (e.g., 600s)
