No, it's not about CDI bean startup timing.

The reason for `Empty` mode is **security** — this is a server-side web app handling multiple concurrent users. From the Java SDK's `CopilotClientMode.EMPTY` Javadoc:

> "Disables optional features by default. The app must explicitly opt into anything it needs. Required for any scenario where CLI-like ambient behavior is unsafe (e.g., multi-user servers)."

The alternative (`COPILOT_CLI`) enables ambient filesystem access, git operations, memory, and skills — behaviors appropriate for a single-user coding agent but dangerous in a shared server. `Empty` mode enforces deny-by-default so each session only gets the specific tools you explicitly wire up.

The C# demo uses `Empty` mode for the same reason — it has nothing to do with when the object is constructed. The CDI `@Produces`/`@Disposes` pattern is just how Jakarta EE manages the singleton's lifecycle (creation + guaranteed cleanup); the mode choice is orthogonal.

I didn't state "CDI push" — that phrase comes from the existing plan document. But to answer your question:

"CDI push" isn't a standalone technology. It refers to the pattern where `f:websocket` integrates with CDI: you inject a `@Push PushContext` into a CDI bean and call `pushContext.send(data)` from server-side code. The CDI container manages the `PushContext` injection and ensures the WebSocket channel is correctly resolved.

The flow is:

1. XHTML declares `<f:websocket channel="agentUpdates" onmessage="handleUpdate" />`
2. A CDI bean injects `@Inject @Push(channel = "agentUpdates") PushContext pushContext;`
3. From any thread (including the virtual thread running the agent), call `pushContext.send(jsonPayload)`
4. The Faces runtime serializes and delivers it to connected clients

So the spike question "Confirm OpenLiberty 26.0.0.5 `faces-4.0` feature supports `f:websocket` with CDI push" is asking: does the OpenLiberty Faces 4.0 implementation wire up the `@Push PushContext` CDI injection correctly? Want me to research that now?
